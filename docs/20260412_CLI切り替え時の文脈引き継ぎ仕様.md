# CLI切り替え時の文脈引き継ぎ仕様

## 対象範囲

- 同一ペイン内で Codex / GitHub Copilot / Gemini を切り替えるときの会話履歴、native session、補助コンテキストの扱いを対象にする。
- 異なるペイン間の共有ドックによる共有は対象外。ただし「ユーザー入力本文と補助情報を混ぜない」という原則は共通とする。

## 基本方針

- 正本は各ペインが持つ会話履歴とする。
- CLI は切り替え可能な実行先とする。
- native session は、各CLIが内部で持つ補助状態として扱う。
- ユーザー入力本文は原則そのまま渡す。
- ツール側が追加する情報は、本文に自然文として混ぜず、短い補助コンテキストとして分離する。
- 補助コンテキストは事実ベースに限定し、推測や依頼範囲の拡張を入れない。
- 同一ペインの会話文脈として必要な場合だけ、直近会話を小さく渡す。
- CLI切り替えなどで未同期差分がある場合は、その未同期差分を優先して渡す。
- 同じCLIで会話を続ける場合でも、native session だけに頼り切らず、直近の user / assistant 会話を小さく渡す。
- 「上記」「先ほど」などの参照語がある場合は、同一ペイン補助コンテキストを優先して参照先を解決する。
- 参照語がない通常の会話でも、前後関係の把握に必要な範囲で同一ペイン補助コンテキストを使ってよい。
- ただし、ユーザー入力を再解釈したり、依頼範囲を広げたりしない。

## 用語

| 用語 | この文書での意味 |
| --- | --- |
| ペイン履歴 | ペインが持つ user / assistant / system の会話履歴。正本。 |
| provider | Codex / GitHub Copilot / Gemini などのCLI種別。 |
| native session | 各CLIの内部session。ペイン履歴とは別の補助状態。 |
| session scope | ペイン、provider、model、作業対象パスなどから作る session の適用範囲。 |
| sync point | その provider に最後に渡したペイン履歴位置。まずは `lastSharedLogEntryId` で扱う。 |
| 未同期差分 | sync point より後に発生した、その provider がまだ見ていない会話。 |
| 補助コンテキスト | 未同期差分、または同一ペインの直近会話から抽出して、ユーザー入力本文とは別に渡す参考情報。命令ではない。 |

## 期待する切り替え動作

### 1. 同じCLIで会話を続ける

例:

- Codexで話す
- そのままCodexへ次の指示を出す

期待動作:

- Codex の native session を再利用する。
- native session は再利用する。
- ただし native session だけに頼り切らず、直近の user / assistant 会話を小さく補助コンテキストとして渡す。
- ペイン履歴には通常どおり user / assistant を追加する。

### 2. 別CLIへ初めて切り替える

例:

- Codexで話す
- Geminiへ切り替える
- Geminiへ「上記を確認して」と聞く

期待動作:

- Gemini 側に同じ scope の native session がなければ新規 native session を開始する。
- Codex側の直近会話を、短い補助コンテキストとして Gemini に渡す。
- 補助コンテキストは「参考情報」であり、ユーザー入力本文より優先しない。
- 参照語がない場合でも、会話の前後関係に必要な範囲では同一ペインの文脈として扱う。

### 3. 別CLIで会話した後、元のCLIへ戻る

例:

- Codexで話す
- Geminiに切り替える
- Geminiで会話する
- Codexに戻る

期待動作:

- Codex 側に同じ scope の native session が残っていれば再利用する。
- ただし Codex は Gemini での会話をまだ見ていないため、その未同期差分を補助コンテキストとして渡す。
- native session の再利用と、未同期差分の補助コンテキスト注入は両立する。

このケースで「未同期差分があるから Codex の native session を必ず捨てる」とはしない。

### 4. native session の整合が怪しい場合

新規 native session に切り替えてよいケース:

- session scope が変わった。
- sessionId が空、または復元できない。
- CLI側の session が明確に壊れている。
- 直前の応答が、別件の古い文脈に強く引っ張られたことが検出・確認された。

ただし、最後のケースは自動判定が難しいため、第一段階では「自動で捨てる」よりも、ログで原因を見えるようにすることを優先する。

## 補助コンテキストの粒度

第一段階では以下を基本にする。

- 対象 provider の sync point 以降の user / assistant を見る。
- 未同期差分があれば、それを優先して末尾の最大4件程度を渡す。
- 未同期差分がなければ、同一ペインの直近 user / assistant の最大2件程度を渡す。
- 長くなりすぎないよう、件数と文字数を抑える。
- systemログや streamログは原則渡さない。
- provider、model、時刻などの事実メタ情報は付けてよい。

補助コンテキストに入れないもの:

- ツール側の推測。
- 会話要約の断定的な再構成。
- ユーザーが今回入力していない依頼の追加。
- 他ペインの共有情報の自動混入。

## 実装方式

現在の実装は、完全な意味理解ではなく、ルールベースの抽出とCLIへの明示指示を組み合わせる。

アプリ側でルールベースに行うこと:

- provider ごとに `lastSharedLogEntryId` を sync point として持つ。
- 対象 provider の sync point より後ろにあるペイン履歴を未同期差分として見る。
- 未同期差分から user / assistant のみを抽出する。
- 未同期差分がある場合は、末尾の最大4件だけを補助コンテキストにする。
- 未同期差分がない場合も、native session だけに頼らず、同一ペインの直近 user / assistant 最大2件を補助コンテキストにする。
- session scope が一致する native session は再利用する。
- session scope が違う、または sessionId がない場合は新規 native session にする。
- ユーザー入力本文を書き換えない。
- コマンド確認とストリームログでは、ユーザー入力と補助コンテキストを別項目として表示する。

アプリ側で行っていないこと:

- ユーザー入力が自己完結しているかどうかの意味判定。
- 補助コンテキストとユーザー入力が矛盾しているかどうかの意味判定。
- 過去会話の自動要約。
- 「おそらくこういう意図だろう」という補完。
- 他ペインやCLI外部履歴からの自動探索。

CLI/LLMへの明示指示で行うこと:

- `Pane Context` を、このツールの同一ペイン内の直近会話文脈として渡す。
- 必要な場合は、この `Pane Context` を会話の主たる文脈として使うように指示する。
- `上記` / `先ほど` / `this issue` / `above` などは、`Pane Context` から解決するように指示する。
- ユーザー入力が自己完結している場合、または補助コンテキストと矛盾する場合は、ユーザー入力を優先するように指示する。
- 無関係なCLI記憶、グローバル履歴、推測したリポジトリ課題から文脈を作らないように指示する。

つまり、補助コンテキストの候補を選ぶ部分はルールベースで決まる。
一方で、「この文脈をどの程度使うか」は、現時点ではCLI/LLM側の指示追従に依存する。

このため、完全に決定論的に「自己完結なら補助コンテキストを無視する」とまでは保証しない。
必要であれば将来的に、参照語検出や「補助コンテキストを付ける / 付けない」の明示トグルを追加する余地がある。

## コマンド確認で見せる情報

コマンド確認では、ユーザーが何をCLIに渡そうとしているかを確認できるようにする。

- 実行先CLIとモデル。
- 作業対象。
- native session の再利用有無。
- ユーザー入力本文。
- 同一ペイン補助コンテキスト。
- ペイン間共有コンテキスト。
- 添付画像。
- 実行コマンド。
- 実際にCLIへ渡す入力。

プロンプト未入力でも、ユーザー入力本文以外の情報は表示できるため、コマンド確認は開けるようにする。
その場合、ユーザー入力本文の欄は「未入力」と表示するが、実際のCLI入力に「未入力」という文字列を混ぜない。

## ストリームログで見せる情報

解析しやすくするため、ストリームログには以下を残す。

- CLI切り替えイベント。
- 切り替え元 / 切り替え先。
- モデル。
- native session を再利用したか、新規開始したか。
- 実行時に渡した主要情報の構造化表示。

CLIを切り替えても、同一ペインのストリームログを消さない。

## 現在実装の方針

2026-04-12 時点の実装では、次の方針にする。

- session scope が一致している native session は原則再利用する。
- 未同期差分がある場合でも、native session は捨てず、補助コンテキストとして差分を渡す。
- 未同期差分がない場合でも、同じCLIでの継続会話が噛み合うよう、直近会話を小さく補助コンテキストとして渡す。
- session scope が違う、sessionId がない、復元できない場合は新規 session にする。
- 「上記」「先ほど」などの参照語は、同一ペイン補助コンテキストがある場合、その内容を参照先として扱うように明示する。
- 参照語がない通常の会話でも、同一ペイン補助コンテキストは必要時の会話文脈として扱う。
- 参照語を、同一ペイン外のCLI記憶、グローバル履歴、推測したリポジトリ課題から解決しないようにする。
- ユーザー入力が自己完結している場合や補助コンテキストと矛盾する場合は、ユーザー入力を優先する。
- 古い文脈に引っ張られた疑いは、まずストリームログとコマンド確認で見える化する。
- 将来的に必要なら、ユーザー操作で「このCLIだけ新規sessionで開始」を追加する。

この方針は、ユーザー入力の意味を変えず、補助コンテキストを分離するというコンセプトとも整合する。
