# Web版パフォーマンスとバックグラウンド実行

## 現象

- 会話やストリームログが増えると、入力欄の入力が引っかかる。
- 複数ペインで同時にCLIを動かすと、画面更新が重くなる懸念がある。
- ブラウザのタブ切り替えや非表示化で、ストリーム接続が切れて処理が中断したように見えることがある。

## 原因

主な原因は Web であること自体より、現在の実装が細かい stream event をそのまま React state と永続化へ流していたこと。

- CLI は `status` / `tool` / `stderr` / `assistant-delta` などのイベントを細かく送る。
- GitHub Copilot や Gemini は `assistant.reasoning_delta` のような高頻度 status を大量に出すことがある。
- そのたびに pane state が更新され、画面全体の再描画と localStorage 保存予約が走る。
- ログモーダルを開いていなくても、会話履歴やストリーム履歴のコピー用テキストを毎回組み立てていた。
- `/api/run/stream` は、ブラウザ接続が close されたときにCLI実行自体も止める実装だった。

## 方針

- Web版でも改善余地はある。Webだから必ず限界、という状態ではない。
- ただし、複数CLIの長時間・高頻度ストリームを大量に扱う場合は、描画・永続化・通信を分離しないと重くなる。
- 第一段階では、低リスクな軽量化を優先する。
- VS Code拡張化は将来案としては有効だが、Web版でもまず改善できる部分がある。

## 今回の対応

### 1. reasoning_delta の間引き

- `assistant.reasoning_delta` のような高頻度 status は、同じ pane では5秒以内の重複を画面 state に反映しない。
- これにより、意味の薄い内部進捗イベントで入力や描画が詰まるのを抑える。
- assistant の実テキスト、stderr、tool、final は維持する。

### 2. localStorage 保存の負荷軽減

- 実行中は pane state の変化が多いため、localStorage 保存の debounce を長めにする。
- 非実行時は短めに保存し、通常操作の復元性は維持する。

### 3. ログモーダル未表示時の計算削減

- 会話履歴・ストリーム履歴の reverse とコピー用テキスト生成を、ログモーダル表示時だけ行う。
- 通常画面で入力しているだけの時に、巨大なログ文字列を毎回組み立てない。

### 4. タブ切り替え時の実行継続

- `/api/run/stream` のブラウザ接続が close されても、サーバー側のCLI実行は止めない。
- 実行結果は短時間サーバー側に保持する。
- 画面へ戻った時、または定期チェック時に `/api/run/status/:paneId` で状態を確認する。
- 完了していれば、結果をペインへ復元する。

## 表示の間引きルール

軽量化は、ユーザーが意味を読み取るための表示を削らない範囲で行う。

間引いてよいもの:

- `assistant.reasoning_delta` のような高頻度で意味の薄い内部進捗。
- ログモーダルを開いていない時の巨大なコピー用テキスト生成。
- 非表示ペインや非表示タブの詳細ログ整形。
- 実行中の localStorage 保存頻度。
- 同じ状態を短時間に繰り返す status 表示。

間引かないもの:

- ユーザー入力中のテキスト、カーソル、選択状態。
- CLI / モデル / 推論レベルなど、ユーザーが直接選ぶ操作UI。
- ドロップダウン、モーダル、ボタンなど、操作中の見た目の安定性。
- assistant の最終回答本文、stderr、エラー、確認待ち、完了、停止。
- コマンド確認に出す「これから渡す情報」の構造化表示。
- 共有情報、同一ペイン補助コンテキスト、添付画像の有無など、実行内容の判断に必要な表示。

今回のモデル選択ドロップダウンでは、装飾背景や再描画負荷よりも操作UIの視認性を優先する。そのため `select` と `option` には不透明な暗色背景を明示し、背面の画像や背景が選択リストへ挟まって見えることを避ける。

## 残る制約

- ブラウザタブが完全に破棄された場合、クライアント側の live stream は当然失われる。
- その場合でもサーバープロセスが生きていれば、最終結果は status API で復元できる。
- サーバー自体が落ちた場合は、実行状態の復元はできない。
- reasoning_delta を間引くため、ストリームログには内部進捗の全イベントは残らない。
- これは解析価値が低い高頻度イベントを落とす判断で、assistant の最終回答や stderr は残す。

## VS Code拡張案

VS Code拡張として作る案は有効。

利点:

- ワークスペース、ファイル、権限、ターミナルとの統合が強い。
- VS Code の通知、Webview、Task、Terminal API を使える。
- 開発者向けの操作として自然。
- 保護パス編集や承認フローをVS Code側の体験に寄せやすい。

懸念:

- Webview を使うなら、UI部分は結局 Web 技術で動く。
- 拡張API、配布、権限、マルチCLI実行管理の設計が別途必要。
- ブラウザ版とVS Code版の二重保守になりやすい。

結論:

- 将来の方向性としては検討価値がある。
- ただし、現在の入力引っかかりやタブ切り替え問題は Web版でも改善できるため、まずは Web版のイベント処理・描画・永続化を軽くする。

## 次の改善候補

- stream event を pane ごとにバッファリングし、一定間隔でまとめて React state に反映する。
- 表示中でない pane の詳細ログ描画をさらに遅延する。
- streamEntries を `system / stderr / tool` と高頻度 `status` で別管理する。
- localStorage ではなく IndexedDB に履歴を逃がす。
- 長い streamEntries を仮想リスト化する。
- サーバー側で run event をリングバッファに保持し、クライアント再接続時に差分取得できるようにする。
