# 保護パス編集と標準エラー警告の扱い

## 現象

- Codex を標準モードで実行すると、workspace を正しく選んでいても .agents / .codex / .git への編集が拒否されることがある。
- 実行ログに stderr（標準エラー出力）が出ていても、最終表示が completed のまま終わるケースがあり、見落としやすかった。

## 用語の補足

- stderr: 標準エラー出力。CLI が警告や失敗理由を流すための別経路の出力。
- generic stderr: 個別分類が付いていない標準エラー。既知パターンとしてはまだ説明文を付けていない状態。
- warningMessage: final イベントに載せる最終注意メッセージ。
- issue summary: 既知パターンに当てはまったときの要約結果。

## 原因

- Codex の標準モードは workspace-write 相当で、workspace 直下でも .agents / .codex / .git は保護対象になる。
- このアプリの実行モデルは one-shot subprocess で、途中の permission request や approval を受けて返答する経路を持たない。
- そのため VS Code のような承認ダイアログ連携は、そのままでは再現できない。

## このリポジトリでの方針

- Codex だけを深く特殊実装して IDE 相当の承認フローを持ち込まない。
- 保護パスへの書き込み失敗は、pane ヘッダー直下に短い notice を出して気づけるようにする。
- .agents / .codex / .git の編集は、このアプリ内で無理に通そうとせず VS Code へ切り替える運用を優先する。
- stderr（標準エラー出力）に意味のある内容が出た場合は completed にせず attention 扱いにして、Run Log の確認を促す。

## 実装メモ

- サーバー側で final イベントに warningMessage（最終注意メッセージ） / warningStatusText を載せる。
- フロント側では provider 固有の既知エラーを issue summary（既知パターンの要約結果）として要約し、最後の warning とあわせて pane の notice に出す。
- generic stderr（個別分類が付いていない標準エラー）でも、エラー語句や複数行の stderr を検出した場合は attention に寄せる。

## 今後の拡張余地

- 本当に VS Code 相当の承認フローが必要なら、stdin を開いたままにするだけでは足りない。
- 承認要求イベント、応答 API、run registry の双方向化を含むセッションプロトコル設計が必要になる。
- その変更は provider 共通の抽象化まで含めて再設計する前提で扱うべきで、今回の軽量 warning 方針とは分けて考える。
