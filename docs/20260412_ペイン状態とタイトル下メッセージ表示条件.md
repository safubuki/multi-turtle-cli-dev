# ペイン状態とタイトル下メッセージ表示条件

## 対象範囲

- この文書は AI 実行ペインの run / stream に関する状態遷移と、ヘッダーのタイトル下に出るメッセージを対象にする。
- SSH 接続確認、VS Code 起動、Explorer 起動、フォルダ作成などの background action は対象外とする。

## 用語の補足

| 用語 | この文書での意味 |
| --- | --- |
| stderr | 標準エラー出力。CLI が警告や失敗理由を流すための別経路のテキスト。 |
| generic stderr | 個別の原因分類が付いていない標準エラー。Gemini quota や Codex 保護パスのような既知パターンにはまだ当てはまっていない stderr。 |
| meaningful stderr | 無視しにくい標準エラー。行数が多い、error や denied などの語を含む、といった理由で attention に寄せる対象。 |
| issue summary | 既知パターンに当てはまったときの要約結果。UI では短い状態文言と説明文に分かれる。 |
| statusHint | サーバーが final イベントで返す最終状態の判定結果。completed / attention / error のどれか。 |
| warningMessage | final イベントに載る最終注意メッセージ。ヘッダー下 notice にも使う。 |
| notice | タイトル下に出る補足メッセージ欄。warning と error の 2 系統がある。 |

## 1. ペイン状態の意味

| 内部状態 | 画面上の基本ラベル | 主な到達条件 | 主な解除条件 |
| --- | --- | --- | --- |
| running | 実行中 | 実行開始時に設定される。stream の session / assistant-delta / status / tool / stderr を受けている間も、controller が生きていれば維持される。 | final / error / stop / stream 切断で別状態へ遷移する。 |
| completed | 完了 | final イベントの statusHint（最終状態の判定結果）が completed のときに設定される。 | 次の実行開始、または画面再描画上は 60 秒後に案内文へ切り替わる。 |
| attention | 入力 / 確認待ち | final イベントの statusHint（最終状態の判定結果）が attention のとき、既知 issue summary（既知パターンの要約結果）が attention を返したとき、stream 切断時のフォールバック、停止時に設定される。 | 次の実行開始、または別の completed / error への遷移。 |
| error | エラー | error イベントで provider 固有の attention へ落とし込めない失敗が来たときに設定される。 | 次の実行開始。 |

## 2. 状態ごとの具体的な条件

### 実行中

- 実行ボタン押下直後に status は running、statusText は 実行中 になる。
- assistant-delta を受けると status は running のまま、statusText は 応答を生成中 になる。
- status / tool / stderr を受けても controller が有効なら running を維持する。
- 既知の stderr を検出した場合だけ、running 中でも statusText は provider 固有文言へ差し替わる。

### 完了

- サーバーが final を返し、その statusHint（最終状態の判定結果）が completed のときに completed になる。
- 現在の completed 条件は、応答本文があり、追加承認を要求する応答語句がなく、policy denied 系 stderr がなく、meaningful stderr（無視しにくい標準エラー）も残っていないケース。
- completed になったときは lastError を null に戻す。

### 確認待ち

- サーバー final の statusHint（最終状態の判定結果）が attention になる主な条件:
  - 応答本文に approval / confirm / yes/no などの追加確認語句が含まれる。
  - stderr に tool execution denied by policy や Gemini Plan Mode 系文言が含まれる。
  - stderr が generic stderr（個別分類されていない標準エラー）でも meaningful stderr（無視しにくい標準エラー）と判定される。
- フロント側で attention になる主な条件:
  - Gemini の quota 到達。
  - Gemini の Plan Mode 停止。
  - Codex の .agents / .codex / .git など保護パス書き込み拒否。
  - Codex の blocked by policy。
  - stream 接続が途中で切れた。
  - ユーザーが停止した、または停止対象が見つからなかった。

### エラー

- stream の error イベントで issue summary（既知パターンの要約結果）が返らない失敗は error になる。
- AI 実行系では、final の statusHint（最終状態の判定結果）が error になる分岐はサーバー側にあるが、response が空の時点で run 自体が reject されるため、実運用上は error イベント経由が中心になる。
- error では lastError に失敗メッセージが入り、ヘッダー notice は赤系になる。

## 3. タイトル下 1 行目の表示規則

タイトル入力の直下にある 1 行目は headerStatusText で、次の優先順で決まる。

| 優先順 | 表示内容 | 条件 |
| --- | --- | --- |
| 1 | 応答待ちが長くなっています | runInProgress が true で、lastActivityAt から 120 秒超更新がない。 |
| 2 | pane.statusText | status が completed かつ、lastFinishedAt から 60 秒以内。通常は 完了。 |
| 3 | 指示または画像を追加してください | status が idle、または completed から 60 秒超で、過去の run 履歴がない。 |
| 4 | 次の指示または画像を追加してください | status が idle、または completed から 60 秒超で、過去の run 履歴がある。 |
| 5 | pane.statusText | 上記以外。running / attention / error / updating の通常表示。 |

## 4. pane.statusText の主な発生源

| 表示文字列 | 主な設定タイミング |
| --- | --- |
| 実行中 | 実行開始直後 |
| 応答を生成中 | assistant-delta 受信時 |
| 完了 | final completed |
| 入力 / 確認待ち | attention だが個別文言がない場合の既定値 |
| 確認待ちです | final attention で承認要求語句を検出した場合 |
| ログ確認が必要です | final attention で meaningful stderr（無視しにくい標準エラー）を検出した場合 |
| Gemini の利用上限に達しました | Gemini quota 系 issue summary（既知パターンの要約結果） |
| Gemini が Plan Mode で停止しました | Gemini Plan Mode issue summary（既知パターンの要約結果） |
| Codex が保護パスを書き込めません | Codex 保護パス issue summary（既知パターンの要約結果） |
| Windows が書き込みを拒否しました | Windows + 保護パス issue summary（既知パターンの要約結果） |
| Codex のツール実行がポリシーで拒否されました | Codex policy deny issue summary（既知パターンの要約結果） |
| 停止要求を送信中 | Stop 実行直後 |
| 停止しました | 停止完了 |
| エラー | error イベントで issue summary（既知パターンの要約結果）がない場合 |

## 5. タイトル下 notice（補足メッセージ）の表示種別

タイトル下の 2 段目 notice（補足メッセージ）は headerNoticeText = pane.lastError?.trim() のときだけ表示される。

| 種別 | スタイル | 表示条件 | 主な文面ソース |
| --- | --- | --- | --- |
| 非表示 | なし | lastError が null または空文字 | なし |
| Warning notice | 黄系 | lastError があり、status が error ではない | issue summary の displayMessage、final の warningMessage（最終注意メッセージ）、stream 切断時の displayMessage |
| Error notice | 赤系 | lastError があり、status が error | error イベントの message、または error 扱いの失敗メッセージ |

## 6. notice が更新されるタイミング

- 実行開始時: lastError は null にリセットされ、notice は消える。
- 実行中の stderr: provider 固有の既知 issue summary（既知パターンの要約結果）に一致した時だけ lastError が入る。generic stderr（個別分類されていない標準エラー）だけではこの時点で notice は出ない。
- final attention: warningMessage（最終注意メッセージ）があれば lastError に入り、notice を出す。warningMessage がなくても、実行中に既に lastError が入っていればそれを維持する。
- final completed: lastError は null になり、notice は消える。
- error: lastError に失敗文面が入り、notice を出す。
- stop 完了: lastError は null に戻る。

## 7. 補足

- 停滞表示は pane.status 自体を attention に変えるのではなく、visualPaneStatus を stalled として見た目だけ attention 系に寄せている。
- completed のヘッダー表示は 60 秒後に案内文へ戻るが、lastError が残っていなければ notice は出ない。
