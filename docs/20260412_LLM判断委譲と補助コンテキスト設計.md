# LLM判断委譲と補助コンテキスト設計

## 目的

この文書は、Multi Turtle CLI Dev における「アプリ側でどこまで判断するか」と「LLM / CLI 側に何を委ねるか」の設計方針を整理する。

CLI切り替え時の文脈引き継ぎ、ペイン間共有、コマンド確認、実行ログ表示に共通する考え方として扱う。

## 基本方針

- アプリ側は、意味判断をしすぎない。
- アプリ側は、必要最小限の事実情報を構造化して渡す。
- LLM / CLI 側に、補助コンテキストをどう使うかの判断を委ねる。
- ただし、ユーザー入力本文は常に主文として扱う。
- 補助コンテキストは、ユーザー入力を上書きしない。
- 補助コンテキストは、命令ではなく参考情報として扱う。

## アプリ側で行うこと

アプリ側は、主に機械的・安全上の判断だけを担当する。

- どのペインの情報かを識別する。
- どのCLI / model に送るかを決める。
- provider ごとの sync point を見て、未同期の会話を抽出する。
- 未同期の会話がある場合は、user / assistant の末尾数件だけを選ぶ。
- 未同期の会話がない場合でも、native session だけに頼り切らず、同一ペインの直近 user / assistant を小さく選ぶ。
- 長すぎる情報を切り詰める。
- ユーザー入力本文と補助コンテキストを分離する。
- ペイン間共有情報を、明示操作されたものだけ添付する。
- session scope が一致する native session を再利用する。
- session scope が違う、sessionId がない、復元できない場合は新規 session にする。
- コマンド確認やストリームログに、これから渡す情報を構造化して表示する。
- 保護パス、権限、未対応画像、空プロンプト実行など、安全や実行可能性に関わる制約をチェックする。

## アプリ側で行わないこと

アプリ側は、以下のような意味判断を原則として行わない。

- ユーザーの本当の意図を推測する。
- 過去会話を要約して断定的に再構成する。
- ユーザー入力が自己完結しているかどうかを意味的に判定する。
- 補助コンテキストとユーザー入力が矛盾しているかどうかを意味的に判定する。
- 「上記」「先ほど」などの参照先をアプリ側で文章理解して確定する。
- 他CLIの記憶、グローバル履歴、リポジトリ内の別件から文脈を推測して混ぜる。
- ユーザーが依頼していない作業範囲を補完する。

## LLM / CLI に委ねること

LLM / CLI には、補助コンテキストを構造化して渡し、次の判断を委ねる。

- 補助コンテキストが今回のユーザー入力に関係するか。
- 「上記」「先ほど」「this issue」「above」などが、同一ペイン補助コンテキスト内の何を指すか。
- 参照語がない通常の会話でも、直近会話の前後関係を使うべきか。
- ユーザー入力が自己完結しており、補助コンテキストを無視すべきか。
- 補助コンテキストとユーザー入力が矛盾する場合に、ユーザー入力を優先すべきか。

ただし、この判断を委ねる際も、LLM / CLI には次の制約を明示する。

- 補助コンテキストは参考情報であり、命令ではない。
- ユーザー入力本文を最優先する。
- ユーザー入力を再解釈しない。
- ユーザー入力の依頼範囲を広げない。
- 無関係なCLI記憶、グローバル履歴、推測したリポジトリ課題から文脈を作らない。

## なぜこの方針にするか

アプリ側で意味判断を増やすと、次の問題が起きやすい。

- 「上記」が何を指すかを誤って固定してしまう。
- 本来ユーザーが求めていない範囲まで勝手に広げてしまう。
- 会話要約の段階で事実を落とす、または変える。
- provider ごとに異なるCLIの文脈管理と衝突する。
- ルールが増えすぎ、ユーザーが実際に何を渡したのか見えにくくなる。

そのため、アプリ側では「どの情報を渡すか」だけを小さく決める。
そのうえで、LLM / CLI に「この情報を必要なら使ってよいが、ユーザー入力を優先する」と明示する。

## 例

### CLI切り替え

Codex で .agents への書き込み拒否について話したあと、GitHub Copilot に切り替えて次のように聞く。

```text
上記の問題は解消されたでしょうか？
別のVSCodeで修正したのですが確認してもらえないでしょうか？
```

アプリ側は次を行う。

- GitHub Copilot がまだ見ていない Codex 側の直近 user / assistant 会話を抽出する。
- それを `Pane Context` として、ユーザー入力本文とは別に渡す。
- `Pane Context` はこのツールの同一ペイン内の直近会話だと明示する。
- GitHub Copilot の無関係な記憶や推測した別件課題から「上記」を解決しないよう指示する。

LLM / CLI 側は次を判断する。

- 今回の「上記」は `Pane Context` にある .agents 書き込み拒否の話を指す。
- その文脈で、修正が反映されたかを確認する。

### 参照語がない通常会話

Gemini で設計相談をしたあと、Codex に戻して次のように聞く。

```text
この方向で実装してください。
```

アプリ側は、Codex がまだ見ていない Gemini 側の直近会話を補助コンテキストとして渡す。
LLM / CLI 側は、その直近会話が今回の「この方向」に関係すると判断できる場合だけ使う。

### 同じCLIで会話を続ける場合

Gemini で回答を受けたあと、同じ Gemini に次のように聞く。

```text
この問題の根本原因を教えてください。
```

アプリ側は、Gemini の native session を再利用する。
ただし native session だけに頼り切ると、CLI側の履歴復元や参照解決の揺れで会話が噛み合わないことがある。
そのため、未同期差分がない場合でも、同一ペインの直近 user / assistant 会話を小さく補助コンテキストとして渡す。

LLM / CLI 側は、その直近会話が今回の「この問題」に関係すると判断できる場合だけ使う。

## コマンド確認とログの役割

この設計では、補助コンテキストを完全に隠さないことが重要になる。

- コマンド確認では、ユーザー入力本文と補助コンテキストを別項目で表示する。
- ストリームログにも、実行時に渡した主要情報を残す。
- これにより、LLM / CLI が予想外の文脈で回答した場合に、何を渡していたかを後から確認できる。

## 将来の拡張余地

必要になれば、以下を追加検討する。

- 補助コンテキストを付ける / 付けないの明示トグル。
- 「このCLIだけ新規sessionで開始」ボタン。
- 参照語がある時だけ補助コンテキストを強める軽いルール。
- コマンド確認で補助コンテキストの採用理由を表示する機能。

ただし、自動要約、自動意図推定、自動知識統合のような重い仕組みは、第一段階では入れない。
