# TAKO シンプル化 + Claude対応 仕様書

## 目的

TAKOを、複数ペインで大量のAI CLIを同時運用するツールではなく、
**1つのワークスペース内で、複数AI CLIを切り替えながら開発作業を引き継ぐためのシンプルなAI CLIハンドオーバーコンソール**として再定義する。

主な目的は以下。

- Codex / Claude / Copilot / Gemini を1つのworkspace内で切り替えて使えるようにする
- あるAIで進めた作業内容を、別AIへ最小限の文脈で引き継げるようにする
- 過剰なシステムプロンプトや長い会話履歴注入を避ける
- 各CLIのネイティブな挙動をできるだけ尊重する
- RunTraceとHandoffNoteにより「何を渡して、何が起きたか」を確認できるようにする

## 今回やらないこと

今回の主目的から外すもの。

- Windowsアプリ化
  - 将来検討として1行だけ残す
  - 今回はWeb/React/Node構成のまま進める
- 10個以上のペイン運用
- 複数workspaceの同時管理
- 2x2 / 3列 / Focus などの高度なマルチレイアウト強化
- xterm / PTY による完全ターミナルモード
- 実行キュー / 依存関係 / 自律会議
- VS Code拡張化
- Quartet Hubとの標準統合
  - 将来的に外部ツールからworkspace指定でTAKOを開ける余地は残す

## 新しいTAKOの定義

TAKOは、1つの開発ワークスペース内で、
Codex / Claude / Copilot / Gemini などのAI CLIを切り替えながら開発作業を継続するための
**AI CLI Handoff Console** である。

TAKOは複数ワークスペースの配置やウィンドウ管理を主目的にしない。
その領域はQuartet Hubなどの外部ツールが担当できる。

TAKOの中核価値は以下。

1. workspace固定
2. AI CLI切り替え
3. native session管理
4. Handoff Note作成
5. RunTrace保存
6. 最小限の文脈注入
7. provider間の安全な作業引き継ぎ

## 対応Provider

現状のproviderにClaudeを追加する。

```ts
type ProviderId = 'codex' | 'copilot' | 'gemini' | 'claude'
```

想定インストール状態:

```text
Codex CLI:        npm i -g @openai/codex
GitHub Copilot:  npm install -g @github/copilot
Gemini CLI:      npm install -g @google/gemini-cli
Claude Code:     curl -fsSL https://claude.ai/install.cmd -o install.cmd && install.cmd && del install.cmd
```

Claude Codeはnpm package前提にしない。
`claude` コマンドの存在確認で検出する。

## Provider検出仕様

既存のnpm package前提のprovider catalogを拡張し、providerごとにinstall kindを持たせる。

```ts
type ProviderInstallKind =
  | 'npm-global'
  | 'official-installer'
  | 'path-command'

type ProviderRuntimeInfo = {
  provider: ProviderId
  label: string
  installKind: ProviderInstallKind
  available: boolean
  commandPath: string | null
  installedVersion: string | null
  latestVersion: string | null
  updateAvailable: boolean
  updateCommand: string | null
  error: string | null
}
```

provider別方針:

```ts
codex: {
  installKind: 'npm-global',
  packageName: '@openai/codex',
  commandNames: ['codex']
}

copilot: {
  installKind: 'npm-global',
  packageName: '@github/copilot',
  commandNames: ['copilot']
}

gemini: {
  installKind: 'npm-global',
  packageName: '@google/gemini-cli',
  commandNames: ['gemini']
}

claude: {
  installKind: 'official-installer',
  packageName: null,
  commandNames: ['claude']
}
```

Claude検出では以下を行う。

- Windowsでは `where claude` 相当で探索
- macOS/Linuxでは `command -v claude` 相当で探索
- `claude --version` または `claude --help` で実行可能性を確認
- npm packageが見つからないことをClaudeのエラーにしない
- Claude未検出時は公式インストーラで入れる旨を表示する

## 実行モード

TAKOは、毎回長い補助コンテキストを自動注入しない。

以下のモードを導入する。

```ts
type PromptInjectionMode =
  | 'native-minimal'
  | 'structured-context'

type ContextPolicy =
  | 'off'
  | 'handoff-note-only'
  | 'recent-pane-context'
  | 'full-structured'
```

### native-minimal

デフォルト。

- ユーザー入力本文をできるだけそのままCLIへ渡す
- workspaceはcwdやCLI引数で指定する
- Handoff Noteが選択されていなければ、余計な前置きを付けない
- 各CLIのネイティブな挙動を尊重する

### structured-context

必要時のみ使う。

- 同一ペイン補助コンテキスト
- 共有コンテキスト
- Handoff Note
- 添付画像情報
- workspace情報

などを構造化して渡す。

ただし、ユーザー入力本文は常に最優先とする。

## ContextPolicyのデフォルト

| 状況                            | デフォルト                      |
| ------------------------------- | ------------------------------- |
| 同じproviderで継続              | `off`                           |
| 別providerへ切替                | `handoff-note-only`             |
| ユーザーが「上記」「先ほど」などを使う | `recent-pane-context` を提案   |
| 詳細レビューや調査依頼          | `full-structured` を選択可能    |
| ユーザーが明示的に文脈不要とした場合 | `off`                      |

## Handoff Note

provider間の引き継ぎは、会話履歴全文ではなく短いHandoff Noteを基本にする。

```ts
type HandoffNote = {
  id: string
  sourceProvider: ProviderId
  targetProvider?: ProviderId
  workspaceLabel: string
  taskGoal: string
  currentState: string
  decisions: string[]
  changedFiles: string[]
  knownIssues: string[]
  nextSuggestedAction: string
  sourceRunId?: string
  createdAt: number
}
```

Handoff Noteの例:

```markdown
# Handoff Note

## Goal
Android preview の動画境界カクつきを改善する。

## Current State
Codexで境界処理の調査と一部修正を実施済み。
buildは成功。実機確認は未完了。

## Decisions
- previewの通常再生中に過剰なseekを入れない
- 境界補正より状態管理を優先する
- iOS Safari / export へ影響させない

## Changed Files
- src/...
- docs/...

## Known Issues
- Android実機で微小な引っかかりが残る可能性
- 旧boundary smoothing系の残存確認が必要

## Next Suggested Action
差分を確認し、不要なseekや古い補正処理が残っていないかレビューしてください。
```

## RunTrace

1回の実行ごとにRunTraceを保存する。

```ts
type RunTrace = {
  runId: string
  paneId: string
  provider: ProviderId
  model: string
  workspaceLabel: string
  workspacePath: string
  promptInjectionMode: PromptInjectionMode
  contextPolicy: ContextPolicy
  sessionId: string | null
  sessionScopeKey: string | null
  prompt: string
  handoffNoteId?: string
  attachedContextIds: string[]
  imageAttachmentIds: string[]
  commandLinePreview?: string
  status: 'running' | 'completed' | 'attention' | 'error' | 'stopped'
  responseSummary?: string
  warningMessage?: string
  errorMessage?: string
  createdAt: number
  finishedAt?: number
}
```

RunTraceの目的:

- どのAIに何を渡したか確認する
- Handoff Noteの元情報にする
- AIの回答がおかしい時に、プロンプト注入・session再利用・context混入を切り分ける
- 後から「この作業はどのAIが何をしたか」を確認する

## UI方針

### 基本画面

```text

[Provider Switch]
```

- provider切替
- workspace表示

- 停止
- コマンド確認
- Handoff Note生成
- Handoff Noteを別providerへ渡す
- RunTrace確認
- session reset
- 今回だけ新規session

## セッション管理

### 原則

- 同じproviderで続ける場合はnative sessionを再利用する
- providerを切り替える場合、native sessionは共有しない
- provider間の引き継ぎはHandoff Noteを使う
- sessionが壊れた疑いがある場合は、ユーザー操作で「このproviderだけ新規session」を選べるようにする

### 追加UI

- `このproviderだけ新規sessionで開始`
- `現在providerのsessionをリセット`
- `Handoff Noteだけ渡して開始`
- `文脈なしで開始`

## Claude対応の最小実装

初期Claude対応では、完璧なstream parserやresumeまでは狙わない。

### 初期対応範囲

- provider一覧にClaudeを追加
- `claude` command検出
- Claude未検出時の説明表示
- Claudeへpromptを渡して実行
- stdout/stderrをTAKOの共通イベントへ変換
- 最終応答をOutputに表示
- RunTraceに記録
- Handoff NoteをClaudeへ渡せる

### 初期対応で無理にやらないこと

- Claude native session resume完全対応
- Claudeの内部JSON stream完全解析
- Claude固有の全permission mode対応
- Claude hooks / MCP / subagent統合
- Claude設定ファイルの編集支援

## 実装計画

### Phase 1: 仕様と型の整理

1. `ProviderId` に `claude` を追加
2. `PromptInjectionMode` を追加
3. `ContextPolicy` を追加
4. `RunTrace` 型を追加
5. `HandoffNote` 型を追加
6. provider install kind / runtime info 型を追加

受け入れ条件:

- TypeScript buildが通る
- 既存providerの型エラーが出ない
- Claude追加後も既存Codex/Copilot/GeminiのUIが壊れない

### Phase 2: Provider検出の拡張

1. npm global providerとofficial installer providerを分離
2. `claude` command検出を追加
3. Claude未検出時の表示を追加
4. npm package missingをClaudeのエラーにしない

受け入れ条件:

- Codex/Copilot/Geminiは従来どおり検出される
- Claudeは `claude` commandがあればavailableになる
- Claudeが未検出でもアプリ全体は起動する
- provider catalog APIが4provider分返す

### Phase 3: Claude実行の最小対応

1. `cliRunner.ts` にClaude分岐を追加
2. Claudeはまず `native-minimal` でpromptを渡す
3. stdout/stderrを共通RunStreamEventへ変換
4. final responseを返す
5. Claude実行時もstopできるようにする

受け入れ条件:

- Claudeを選択して実行できる
- 応答がOutputに表示される
- stderrがActivityに表示される
- stopでClaudeプロセスを停止できる
- Codex/Copilot/Geminiの挙動が変わらない

### Phase 4: Handoff Note

1. Handoff Note生成ロジックを追加
2. 直近RunTraceまたは最新応答からHandoff Noteを作る
3. ユーザーがHandoff Noteを確認・編集できる
4. 「Claudeへ引き継ぐ」「Codexへ戻す」などの導線を追加
5. Handoff Noteをpromptに添付して実行できる

受け入れ条件:

- 最新応答からHandoff Noteを作れる
- Handoff Noteを編集できる
- 別providerへHandoff Noteを渡せる
- Handoff Noteなし実行もできる
- Handoff Noteは命令ではなく参考情報として渡される

### Phase 5: RunTrace

1. 実行開始時にRunTraceを作成
2. 実行完了/エラー/停止時にRunTraceを更新
3. RunTraceをUIで確認できる
4. RunTraceからHandoff Noteを作れる
5. localStorageまたは既存永続化層に保存する

受け入れ条件:

- 実行ごとにRunTraceが残る
- provider/model/workspace/prompt/contextPolicy/sessionId/statusが確認できる
- エラー時にもRunTraceが残る
- Handoff Note生成元RunTraceを辿れる

### Phase 6: ContextPolicyの実装

1. デフォルトを `native-minimal` + `off` に寄せる
2. provider切替時だけ `handoff-note-only` を提案
3. `recent-pane-context` は明示選択または参照語がある場合だけ使う
4. `full-structured` は詳細レビュー用の明示モードにする
5. コマンド確認で実際に渡す内容を表示する

受け入れ条件:

- 毎回長いPane Contextが勝手に混ざらない
- Handoff Noteだけ渡す実行ができる
- 文脈なし実行ができる
- structured context実行も明示的には可能
- コマンド確認でprompt注入内容が確認できる

### Phase 7: シンプル化

1. 多ペイン前提のUIを目立たないようにする
2. workspace固定・provider切替・handoff中心の画面へ整理
3. 既存の2x2/3列/Focusは削除または優先度を下げる
4. 共有コンテキスト機能はHandoff Note中心に置き換える
5. README / docsを新方針へ更新する

受け入れ条件:

- 初見で「1 workspace内でAIを切り替えるツール」と分かる
- 多数ペイン運用を前提にしない
- Handoff NoteとRunTraceが主要導線になっている
- 既存の基本実行機能は壊れない
