## 1. プロダクト定義

### 1.1 目的

- 複数の CLI エージェントを一画面で整然と運用する。
- IDE ほど情報過多にせず、CLI 単体よりも停止検知、切り替え、再利用をしやすくする。
- コードを直接見続けなくても、複数タスクの進行状況と危険信号を判断できるようにする。

### 1.2 プロダクト定義

- Terminator の分割運用に近い体験を、Web ベースのタスクデッキとして再構成する。
- 各ペインは CLI セッション、対象ワークスペース、共有コンテキスト、進行状態を持つ。
- 生 PTY を前提にせず、CLI の品質を維持したまま、操作性だけを UI が補完する。

### 1.3 実装済みコア機能

#### ペイン単位の実行

- Codex CLI / GitHub Copilot CLI / Gemini CLI の切り替え
- モデル選択
- 推論レベル選択
- 実行権限レベル選択
  - Codex は balanced で `--full-auto`、max で `--dangerously-bypass-approvals-and-sandbox` を使う
  - Gemini は auto_edit / yolo 相当の切り替え
  - Copilot は allow-all-tools / allow-all 相当の切り替え
- Codex Fast モード切り替え

#### ワークスペース操作

- ローカルワークスペース選択
- ローカル選択の初期位置は、前回ブラウズ位置、ペインの現在ワークスペース、OS別デフォルトの順で決める
  - Windows は存在するドライブを優先する
  - Linux / macOS は Home を優先する
  - このツール自身のワークスペースは初期位置として自動優先しない
- ネイティブフォルダダイアログからのローカル追加
- VS Code で開く
- SSH ホスト入力
- SSH 先のワークスペース候補検出
- SSH 先にインストール済みの CLI 検出
- Remote-SSH URI で VS Code を開く

#### 会話継続

- 各ペインに CLI セッション ID を保持
- 同一ペインで次回実行時に resume を優先
- ページ再読み込み後も状態を localStorage に保持
- 実行中に切断されたセッションは「前回の実行は切断されました」として可視化

#### 情報引き継ぎ

- ペインの結果を共有コンテキストとして保存
- 他ペインから必要な共有コンテキストを選択して次回実行に添付
- 自動共有トグルで、完了結果を即座に共有ストックへ反映

#### 実行可視化

- ペイン枠と LED の色で状態表示
  - 待機: ニュートラル
  - 実行中: シアン
  - 完了: グリーン
  - 確認待ち: イエロー
  - エラー: レッド
- Live Output にストリーミング応答を表示
- Activity にツール実行、状態変化、stderr を表示
- Conversation にユーザー / アシスタント / システム履歴を保持
- 一定時間更新がない実行を停滞として検知
- ヘッダー直下に warning / error notice を出して確認が必要な実行を見落としにくくする

#### 補助操作

- ペイン複製
- 会話リセット
- 最新応答コピー
- 実行停止
- レイアウト切り替え（2x2 / 3列 / Focus）

### 1.4 バックエンド設計

- `POST /api/run/stream` — NDJSON でイベントを返す（assistant-delta / status / tool / stderr / session / final / error）
- `POST /api/run/stop` — ペイン単位で実行停止
- `POST /api/ssh/workspaces` — SSH 先ワークスペース探索
- `POST /api/ssh/inspect` — SSH 先 CLI 検出
- `POST /api/system/open-vscode` — 対象ワークスペースを VS Code で開く

### 1.5 UI 方針

- 生ターミナルではなく、Composer + Live Output + Activity の三層構成
- 直接ターミナルを触るより、タスクを流し込んで監視する方に最適化
- 停止検知、共有、切り替え、再開を UI が担当し、CLI の回答品質は CLI 本体に委ねる

### 1.6 次の拡張候補（2026年4月時点）

- xterm / PTY を使った完全ターミナルモード
- 共有コンテキストのタグ付けと検索
- ペイングループごとのテンプレート保存
- 実行キューと依存関係
- 差分プレビューや成果物比較ビュー

## 2. AIファーストリファクタリング概要

### 2.1 目的

- `App.tsx` と `TerminalPane.tsx` に集まりやすかった state 更新、effect、実行ハンドラを役割ごとに分離し、読み始める場所と変更すべき場所を判断しやすくする。
- AI 実行、shell 操作、prompt composer、画面効果、永続化/復元の責務境界を明示し、将来の修正で広い範囲を同時に触らなくてよい状態にする。
- 見た目や API は変えずに、回帰を検出しやすいテスト単位へ整理する。

### 2.2 分解範囲

#### App 側（`src/App.tsx` から切り出した責務）

| モジュール | 責務 |
| --- | --- |
| `src/lib/workspaceActions.ts` | ワークスペース選択、ローカル/SSH 操作、VS Code 起動まわり |
| `src/lib/shellActions.ts` | shell 実行と停止、shell 出力の反映 |
| `src/lib/promptImageActions.ts` | prompt 添付画像の追加、削除、リソース解放 |
| `src/lib/sessionContextActions.ts` | ペイン共有コンテキスト、共有ストック、自動共有の同期 |
| `src/lib/runActions.ts` | AI 実行開始、stream 反映、停止、最終状態反映 |
| `src/lib/paneInteractionActions.ts` | ペイン選択、複製、レイアウト操作などの UI 連携 |
| `src/lib/paneLifecycleActions.ts` | ペイン生成、削除、初期化、workspace 反映 |
| `src/lib/paneProviderActions.ts` | provider 切り替え、model / reasoning / approval 設定の同期 |
| `src/lib/panePresentationActions.ts` | タイトル下メッセージ、run 状態に応じた表示更新 |
| `src/lib/paneStateActions.ts` | `updatePane` / `mutatePane` / `appendPaneSystemMessage` などの低レベル state 更新 |
| `src/lib/useAppLifecycle.ts` | bootstrap、localStorage 永続化、focus/visibility 時の refresh、バックグラウンド実行状態の再取得、unmount cleanup |
| `src/lib/useAppUiEffects.ts` | workspace picker 表示中の body scroll lock、Delete キーによる選択ペイン削除、local workspace ペインの自動 browse 読み込み |

#### TerminalPane 側（`src/components/TerminalPane.tsx` から切り出した責務）

| Hook | 責務 |
| --- | --- |
| `useTerminalPanePresentationState.ts` | output modal、run logs modal、preview modal、prompt image preview、copy feedback |
| `useTerminalPanePromptComposer.ts` | prompt draft、textarea 高さ同期、画像 picker / drop / paste、実行直前 flush |
| `useTerminalPaneShellState.ts` | shell 入力、履歴移動、Ctrl+C、Enter 実行、shell modal の expand / close、focus / scroll 復帰 |

`TerminalPane.tsx` に残す責務は、描画構造、props 配線、各 hook から返る値の UI への接続に限定する。

### 2.3 最終的な責務分担

| 層 | 主な責務 |
| --- | --- |
| `src/App.tsx` | ドメイン action / hook の生成と配線、全体レイアウトの組み立て |
| `src/lib/useAppLifecycle.ts` | 復元、保存、フォーカス復帰、バックグラウンド実行状態の追跡、unmount cleanup |
| `src/lib/useAppUiEffects.ts` | App レベルの小粒な UI effect 群 |
| `src/components/TerminalPane.tsx` | ペイン UI の構成とイベント配線 |
| `src/components/useTerminalPanePromptComposer.ts` | 実行前に必要な prompt 入力まわりの状態管理 |
| `src/components/useTerminalPaneShellState.ts` | shell 入力専用の状態とキーボード操作 |
| `src/components/useTerminalPanePresentationState.ts` | modal / preview / copy feedback など表示専用 state |

### 2.4 回帰と修正

分解後、実行開始直後に「停止しました」へ落ち、実行中バーも表示されない回帰が発生した。

**原因:**
- controller を abort する cleanup が、unmount 専用ではなく effect dependency の変化でも走っていた。
- `cleanupAllPromptImageResources` の callback identity が再 render ごとに変わると、rerender 前 cleanup で `controllersRef` が abort された。
- その結果、実際には実行継続可能な run が stop 扱いになり、`runInProgress` もすぐ false に戻っていた。

**修正内容:**
- `cleanupAllPromptImageResources` は ref 経由で最新関数を参照する。
- controller abort を行う cleanup effect は unmount 時だけ実行する。
- `src/test/useAppLifecycle.test.tsx` に、callback identity の変化では abort されず、unmount でのみ abort される回帰テストを追加した。

### 2.5 追加・更新したテスト

- `src/test/useTerminalPanePromptComposer.test.tsx`
- `src/test/useAppLifecycle.test.tsx`
- `src/test/useAppUiEffects.test.tsx`
- `src/test/useTerminalPaneShellState.test.tsx`

### 2.6 検証結果

- `npm test -- --run` で全件成功。
- `npm run build` で成功。
- 既知の非ブロッカーは Vite の chunk size warning のみ。

### 2.7 今後の変更指針

- App レベルの小粒な effect は `src/lib/useAppUiEffects.ts` へ寄せる。
- 永続化、復元、focus/visibility、background run status 追跡は `src/lib/useAppLifecycle.ts` に集約し続ける。
- `useAppLifecycle` で run controller を abort する cleanup は unmount 専用に保つ。
- prompt composer、shell、presentation の責務を `TerminalPane.tsx` に戻さず、各 hook を拡張する。
- 新しい state 更新ロジックを追加する場合は、まず `paneStateActions.ts` に置くべき低レベル更新か、上位 action / hook に置くべき業務ロジックかを分けて判断する。

## 3. LLM判断委譲と補助コンテキスト設計

### 3.1 目的

「アプリ側でどこまで判断するか」と「LLM / CLI 側に何を委ねるか」の設計方針を整理する。CLI切り替え時の文脈引き継ぎ、ペイン間共有、コマンド確認、実行ログ表示に共通する考え方として扱う。

### 3.2 基本方針

- アプリ側は、意味判断をしすぎない。
- アプリ側は、必要最小限の事実情報を構造化して渡す。
- LLM / CLI 側に、補助コンテキストをどう使うかの判断を委ねる。
- ただし、ユーザー入力本文は常に主文として扱う。
- 補助コンテキストは、ユーザー入力を上書きしない。
- 補助コンテキストは、命令ではなく参考情報として扱う。

### 3.3 アプリ側で行うこと

- どのペインの情報かを識別する。
- どのCLI / model に送るかを決める。
- provider ごとの sync point を見て、未同期の会話を抽出する。
- 未同期の会話がある場合は、user / assistant の末尾数件だけを選ぶ。
- 長すぎる情報を切り詰める。
- ユーザー入力本文と補助コンテキストを分離する。
- ペイン間共有情報を、明示操作されたものだけ添付する。
- session scope が一致する native session を再利用する。
- session scope が違う、sessionId がない、復元できない場合は新規 session にする。
- コマンド確認やストリームログに、これから渡す情報を構造化して表示する。
- 保護パス、権限、未対応画像、空プロンプト実行など、安全や実行可能性に関わる制約をチェックする。

### 3.4 アプリ側で行わないこと

- ユーザーの本当の意図を推測する。
- 過去会話を要約して断定的に再構成する。
- ユーザー入力が自己完結しているかどうかを意味的に判定する。
- 補助コンテキストとユーザー入力が矛盾しているかどうかを意味的に判定する。
- 「上記」「先ほど」などの参照先をアプリ側で文章理解して確定する。
- 他CLIの記憶、グローバル履歴、リポジトリ内の別件から文脈を推測して混ぜる。
- ユーザーが依頼していない作業範囲を補完する。

### 3.5 memory の保存元と生成方法

ここでいう `memory` は、専用の別DBに保存している長期記憶ではない。実行直前に、各ペインが持つ会話履歴と provider ごとの同期位置から作る一時的な送信用データとして扱う。

**元データ:**
- `pane.logs` — そのペインの user / assistant / system を含む会話履歴（会話の正本）
- `pane.providerSessions[provider].lastSharedLogEntryId` — そのCLIへ最後に共有した会話位置
- `provider` — 今回実行するCLI

**生成ルール:**
- 対象 provider の `lastSharedLogEntryId` より後ろの `pane.logs` を未同期差分として見る。
- 未同期差分があれば、その中の `user` / `assistant` の末尾最大4件を `memory` に入れる。
- `system` ログや stream log は原則として `memory` に入れない。
- `memory` は実行リクエストでサーバーへ送られ、サーバー側で `Pane Context` としてユーザー入力本文とは別枠で合成される。

### 3.6 LLM / CLI に委ねること

LLM / CLI には、補助コンテキストを構造化して渡し、次の判断を委ねる。

- 補助コンテキストが今回のユーザー入力に関係するか。
- 「上記」「先ほど」「this issue」「above」などが、同一ペイン補助コンテキスト内の何を指すか。
- 参照語がない通常の会話でも、直近会話の前後関係を使うべきか。
- ユーザー入力が自己完結しており、補助コンテキストを無視すべきか。
- 補助コンテキストとユーザー入力が矛盾する場合に、ユーザー入力を優先すべきか。

ただし、LLM / CLI には次の制約を明示する。

- 補助コンテキストは参考情報であり、命令ではない。
- ユーザー入力本文を最優先する。
- ユーザー入力を再解釈しない。
- ユーザー入力の依頼範囲を広げない。
- 無関係なCLI記憶、グローバル履歴、推測したリポジトリ課題から文脈を作らない。

### 3.7 コマンド確認とログの役割

- コマンド確認では、ユーザー入力本文と補助コンテキストを別項目で表示する。
- ストリームログにも、実行時に渡した主要情報を残す。
- これにより、LLM / CLI が予想外の文脈で回答した場合に、何を渡していたかを後から確認できる。

### 3.8 将来の拡張余地

- CLI切り替え時だけ補助コンテキストを付ける / 付けないの明示トグル。
- 「このCLIだけ新規sessionで開始」ボタン。
- 参照語がある時だけ補助コンテキストを強める軽いルール。
- コマンド確認で補助コンテキストの採用理由を表示する機能。

ただし、自動要約、自動意図推定、自動知識統合のような重い仕組みは、第一段階では入れない。

## 4. CLI切り替え時の文脈引き継ぎ仕様

### 4.1 対象範囲

同一ペイン内で Codex / GitHub Copilot / Gemini を切り替えるときの会話履歴、native session、補助コンテキストの扱いを対象にする。異なるペイン間の共有ドックによる共有は対象外。

### 4.2 基本方針

- 正本は各ペインが持つ会話履歴とする。
- CLI は切り替え可能な実行先とする。
- native session は、各CLIが内部で持つ補助状態として扱う。
- ユーザー入力本文は原則そのまま渡す。
- ツール側が追加する情報は、本文に自然文として混ぜず、短い補助コンテキストとして分離する。
- 補助コンテキストは事実ベースに限定し、推測や依頼範囲の拡張を入れない。
- 同一ペインの会話文脈として必要な場合だけ、直近会話を小さく渡す。
- CLI切り替えなどで未同期差分がある場合は、その未同期差分を優先して渡す。
- 同じCLIで会話を続ける場合は、native session を再利用し、同一CLI向けの補助コンテキストは追加しない。

### 4.3 用語

| 用語 | 意味 |
| --- | --- |
| ペイン履歴 | ペインが持つ user / assistant / system の会話履歴。正本。 |
| provider | Codex / GitHub Copilot / Gemini などのCLI種別。 |
| native session | 各CLIの内部session。ペイン履歴とは別の補助状態。 |
| session scope | ペイン、provider、model、作業対象パスなどから作る session の適用範囲。 |
| sync point | その provider に最後に渡したペイン履歴位置。`lastSharedLogEntryId` で管理。 |
| 未同期差分 | sync point より後に発生した、その provider がまだ見ていない会話。 |
| 補助コンテキスト | 未同期差分から抽出して、ユーザー入力本文とは別に渡す参考情報。命令ではない。 |

### 4.4 期待する切り替え動作

#### 同じCLIで会話を続ける

- Codex の native session を再利用する。
- 同一CLI向けの補助コンテキストは追加しない。
- ペイン履歴には通常どおり user / assistant を追加する。

#### 別CLIへ初めて切り替える

- Gemini 側に同じ scope の native session がなければ新規 native session を開始する。
- Codex側の直近会話を、短い補助コンテキストとして Gemini に渡す。
- 補助コンテキストは「参考情報」であり、ユーザー入力本文より優先しない。

#### 別CLIで会話した後、元のCLIへ戻る

- Codex 側に同じ scope の native session が残っていれば再利用する。
- Codex は Gemini での会話をまだ見ていないため、その未同期差分を補助コンテキストとして渡す。
- native session の再利用と、未同期差分の補助コンテキスト注入は両立する。

#### native session の整合が怪しい場合

新規 native session に切り替えてよいケース:

- session scope が変わった。
- sessionId が空、または復元できない。
- CLI側の session が明確に壊れている。
- 直前の応答が、別件の古い文脈に強く引っ張られたことが検出・確認された。

### 4.5 補助コンテキストの粒度

- 対象 provider の sync point 以降の user / assistant を見る。
- 未同期差分があれば、それを優先して末尾の最大4件程度を渡す。
- 長くなりすぎないよう、件数と文字数を抑える。
- systemログや streamログは原則渡さない。
- provider、model、時刻などの事実メタ情報は付けてよい。

補助コンテキストに入れないもの:

- ツール側の推測。
- 会話要約の断定的な再構成。
- ユーザーが今回入力していない依頼の追加。
- 他ペインの共有情報の自動混入。

### 4.6 実装方式

アプリ側でルールベースに行うこと:

- provider ごとに `lastSharedLogEntryId` を sync point として持つ。
- 対象 provider の sync point より後ろにあるペイン履歴を未同期差分として見る。
- 未同期差分から user / assistant のみを抽出する。
- 未同期差分がある場合は、末尾の最大4件だけを補助コンテキストにする。
- session scope が一致する native session は再利用する。
- session scope が違う、または sessionId がない場合は新規 native session にする。
- ユーザー入力本文を書き換えない。

アプリ側で行っていないこと:

- ユーザー入力が自己完結しているかどうかの意味判定。
- 補助コンテキストとユーザー入力が矛盾しているかどうかの意味判定。
- 過去会話の自動要約。

### 4.7 コマンド確認で見せる情報

- 実行先CLIとモデル
- 作業対象
- native session の再利用有無
- ユーザー入力本文
- 同一ペイン補助コンテキスト
- ペイン間共有コンテキスト
- 添付画像
- 実行コマンド
- 実際にCLIへ渡す入力

### 4.8 ストリームログで見せる情報

- CLI切り替えイベント
- 切り替え元 / 切り替え先
- モデル
- native session を再利用したか、新規開始したか
- 実行時に渡した主要情報の構造化表示

## 5. 実行開始確定と補助コンテキスト消費仕様

### 5.1 目的

- 未送信の会話を送信済みとして扱わない。
- ユーザーが明示添付した shared context を、開始失敗で失わない。
- 通常完了とバックグラウンド復元完了で、共有反映の挙動を一致させる。
- shared context と画像説明の送信量を、実運用で扱いやすい大きさに保つ。

### 5.2 用語

| 用語 | 意味 |
| --- | --- |
| 開始準備 | ユーザーが Run を押したあと、サーバーがまだその実行を受理したと確定していない状態。 |
| 開始確定 | サーバーが stream 上で started イベントを返し、フロントが確定できた状態。 |
| 消費確定 | 次回以降の送信候補から外してよいと判断し、内部 state から前進させること。 |
| 完了確定 | 通常の final イベント、またはバックグラウンド復元 completed により、実行結果が確定した状態。 |

### 5.3 基本方針

- 開始確定までは、送信成功を前提にした state 更新をしない。
- pane.logs には、実際に受理された user / assistant 会話だけを残す。
- shared context と prompt 画像は、開始確定までは UI 上に残す。
- 完了処理は、通常完了とバックグラウンド復元完了で共通化する。
- shared context は、明示添付順を尊重しつつ、1 リクエストあたりの送信量を制限する。

### 5.4 実行ライフサイクル

#### 開始準備（Run ボタン押下直後）

この段階で行うこと:

- 実行 payload を組み立てる。
- 実行中の二重起動を防ぐため、ローカル controller を確保する。
- UI 上の status を「実行を開始中」にする。

この段階で行わないこと:

- `pane.logs` への user entry 追加。
- `providerSessions.lastSharedLogEntryId` の更新。
- attached shared context の取り外し。
- shared context store 側の consumed 処理。
- prompt のクリア。
- prompt 画像のクリアと cleanup 予約。

#### 開始確定（started イベント受信時）

サーバーは `/api/run/stream` で `started` イベントを最初に返す。フロントはこれを受けた時だけ実行開始を確定する。

開始確定時に行うこと:

- `pane.logs` に user entry を追加する。
- `providerSessions.lastSharedLogEntryId` をその user entry に進める。
- attached shared context のうち、今回実際に送る分だけを消費確定する。
- prompt をクリアする。
- prompt 画像をクリアし、cleanup を予約する。
- `currentRequestText` と `currentRequestAt` を更新する。
- status を running にし、`runInProgress` を true にする。

#### 開始失敗時の扱い

`started` より前に失敗した場合、その実行は未受理として扱う。

必須動作:

- `pane.logs` に user entry を追加しない。
- `providerSessions.lastSharedLogEntryId` を進めない。
- attached shared context を残す。
- prompt を残す。
- prompt 画像を残す。

#### 完了確定時の扱い

通常の final と、バックグラウンド復元 completed は同じ完了処理を通す。

共通で行うこと:

- assistant 応答を logs と liveOutput に反映する。
- status と lastError を最終結果に合わせて確定する。
- provider session の sessionId と lastSharedLogEntryId を更新する。
- pending share と auto share を確定する。
- workspace refresh をスケジュールする。

### 5.5 Shared Context の送信仕様

`pane.attachedContextIds` の順序を優先して選ぶ。1 リクエストあたりの送信上限:

- 最大 3 件まで送る。
- 各 item の detail は最大 4000 文字まで送る。
- detail の合計は最大 8000 文字まで送る。

上限により今回送られなかった attached context は消費確定しない。次回実行でも引き続き attached のまま残る。

### 5.6 画像説明の責務

画像説明の prompt 追加は provider prompt 組み立て側だけが担当する。共通の combined prompt には `Attached Images` セクションを入れない。これにより、同じ画像説明が prompt に二重挿入されることを防ぐ。

## 6. ペイン状態とタイトル下メッセージ表示条件

### 6.1 対象範囲

AI 実行ペインの run / stream に関する状態遷移と、ヘッダーのタイトル下に出るメッセージを対象にする。SSH 接続確認、VS Code 起動、Explorer 起動、フォルダ作成などの background action は対象外。

### 6.2 用語

| 用語 | 意味 |
| --- | --- |
| stderr | 標準エラー出力。CLI が警告や失敗理由を流すための別経路のテキスト。 |
| generic stderr | 個別の原因分類が付いていない標準エラー。 |
| meaningful stderr | 無視しにくい標準エラー。行数が多い、error や denied などの語を含む等。 |
| issue summary | 既知パターンに当てはまったときの要約結果。 |
| statusHint | サーバーが final イベントで返す最終状態の判定結果（completed / attention / error）。 |
| warningMessage | final イベントに載る最終注意メッセージ。ヘッダー下 notice にも使う。 |
| notice | タイトル下に出る補足メッセージ欄。warning と error の 2 系統がある。 |

### 6.3 ペイン状態の意味

| 内部状態 | 画面上のラベル | 主な到達条件 | 主な解除条件 |
| --- | --- | --- | --- |
| running | 実行中 | 実行開始時に設定。stream イベントを受けている間、controller が生きていれば維持。 | final / error / stop / stream 切断で別状態へ遷移。 |
| completed | 完了 | final の statusHint が completed のとき。 | 次の実行開始、または 60 秒後に案内文へ切り替わる。 |
| attention | 入力 / 確認待ち | final の statusHint が attention のとき、既知 issue summary が attention を返したとき、stream 切断時、停止時。 | 次の実行開始、または別の completed / error への遷移。 |
| error | エラー | error イベントで provider 固有の attention へ落とし込めない失敗が来たとき。 | 次の実行開始。 |

### 6.4 タイトル下 1 行目の表示規則（headerStatusText）

| 優先順 | 表示内容 | 条件 |
| --- | --- | --- |
| 1 | 応答待ちが長くなっています | runInProgress が true で、lastActivityAt から 120 秒超更新がない。 |
| 2 | pane.statusText | status が completed かつ、lastFinishedAt から 60 秒以内。通常は「完了」。 |
| 3 | 指示または画像を追加してください | status が idle または completed から 60 秒超で、過去の run 履歴がない。 |
| 4 | 次の指示または画像を追加してください | status が idle または completed から 60 秒超で、過去の run 履歴がある。 |
| 5 | pane.statusText | 上記以外。running / attention / error / updating の通常表示。 |

### 6.5 pane.statusText の主な発生源

| 表示文字列 | 設定タイミング |
| --- | --- |
| 実行中 | 実行開始直後 |
| 応答を生成中 | assistant-delta 受信時 |
| 完了 | final completed |
| 入力 / 確認待ち | attention だが個別文言がない場合の既定値 |
| 確認待ちです | final attention で承認要求語句を検出した場合 |
| ログ確認が必要です | final attention で meaningful stderr を検出した場合 |
| Gemini の利用上限に達しました | Gemini quota 系 issue summary |
| Gemini が Plan Mode で停止しました | Gemini Plan Mode issue summary |
| Codex が保護パスを書き込めません | Codex 保護パス issue summary |
| Windows が書き込みを拒否しました | Windows + 保護パス issue summary |
| Codex のツール実行がポリシーで拒否されました | Codex policy deny issue summary |
| 停止要求を送信中 | Stop 実行直後 |
| 停止しました | 停止完了 |
| エラー | error イベントで issue summary がない場合 |

### 6.6 タイトル下 notice の表示種別

| 種別 | スタイル | 表示条件 |
| --- | --- | --- |
| 非表示 | なし | lastError が null または空文字 |
| Warning notice | 黄系 | lastError があり、status が error ではない |
| Error notice | 赤系 | lastError があり、status が error |

### 6.7 notice が更新されるタイミング

- **実行開始時:** lastError は null にリセットされ、notice は消える。
- **実行中の stderr:** provider 固有の既知 issue summary に一致した時だけ lastError が入る。generic stderr だけではこの時点で notice は出ない。
- **final attention:** warningMessage があれば lastError に入り、notice を出す。warningMessage がなくても、実行中に既に lastError が入っていればそれを維持する。
- **final completed:** lastError は null になり、notice は消える。
- **error:** lastError に失敗文面が入り、notice を出す。
- **stop 完了:** lastError は null に戻る。

## 7. 保護パス編集と標準エラー警告の扱い

### 7.1 現象

- Codex を標準モードで実行すると、workspace を正しく選んでいても `.agents` / `.codex` / `.git` への編集が拒否されることがある。
- 実行ログに stderr が出ていても、最終表示が completed のまま終わるケースがあり、見落としやすかった。

### 7.2 原因

- Codex の標準モードは workspace-write 相当で、workspace 直下でも `.agents` / `.codex` / `.git` は保護対象になる。
- このアプリの実行モデルは one-shot subprocess で、途中の permission request や approval を受けて返答する経路を持たない。
- そのため VS Code のような承認ダイアログ連携は、そのままでは再現できない。

### 7.3 このリポジトリでの方針

- Codex だけを深く特殊実装して IDE 相当の承認フローを持ち込まない。
- 保護パスへの書き込み失敗は、pane ヘッダー直下に短い notice を出して気づけるようにする。
- `.agents` / `.codex` / `.git` の編集は、このアプリ内で無理に通そうとせず VS Code へ切り替える運用を優先する。
- stderr に意味のある内容が出た場合は completed にせず attention 扱いにして、Run Log の確認を促す。

### 7.4 実装メモ

- サーバー側で final イベントに warningMessage / warningStatusText を載せる。
- フロント側では provider 固有の既知エラーを issue summary として要約し、最後の warning とあわせて pane の notice に出す。
- generic stderr でも、エラー語句や複数行の stderr を検出した場合は attention に寄せる。

### 7.5 今後の拡張余地

本当に VS Code 相当の承認フローが必要なら、stdin を開いたままにするだけでは足りない。承認要求イベント、応答 API、run registry の双方向化を含むセッションプロトコル設計が必要になる。その変更は provider 共通の抽象化まで含めて再設計する前提で扱うべきで、今回の軽量 warning 方針とは分けて考える。

## 8. Web版パフォーマンスとバックグラウンド実行

### 8.1 現象

- 会話やストリームログが増えると、入力欄の入力が引っかかる。
- 複数ペインで同時にCLIを動かすと、画面更新が重くなる懸念がある。
- ブラウザのタブ切り替えや非表示化で、ストリーム接続が切れて処理が中断したように見えることがある。

### 8.2 原因

主な原因は Web であること自体より、細かい stream event をそのまま React state と永続化へ流していたこと。

- CLI は `status` / `tool` / `stderr` / `assistant-delta` などのイベントを細かく送る。
- GitHub Copilot や Gemini は `assistant.reasoning_delta` のような高頻度 status を大量に出すことがある。
- そのたびに pane state が更新され、画面全体の再描画と localStorage 保存予約が走る。
- ログモーダルを開いていなくても、会話履歴やストリーム履歴のコピー用テキストを毎回組み立てていた。
- `/api/run/stream` は、ブラウザ接続が close されたときにCLI実行自体も止める実装だった。

### 8.3 今回の対応

#### reasoning_delta の間引き

- `assistant.reasoning_delta` のような高頻度 status は、同じ pane では5秒以内の重複を画面 state に反映しない。
- assistant の実テキスト、stderr、tool、final は維持する。

#### localStorage 保存の負荷軽減

- 実行中は pane state の変化が多いため、localStorage 保存の debounce を長めにする。
- 非実行時は短めに保存し、通常操作の復元性は維持する。

#### ログモーダル未表示時の計算削減

- 会話履歴・ストリーム履歴の reverse とコピー用テキスト生成を、ログモーダル表示時だけ行う。

#### タブ切り替え時の実行継続

- `/api/run/stream` のブラウザ接続が close されても、サーバー側のCLI実行は止めない。
- 実行結果は短時間サーバー側に保持する。
- 画面へ戻った時、または定期チェック時に `/api/run/status/:paneId` で状態を確認する。
- 完了していれば、結果をペインへ復元する。

#### フロント実装上の注意

- bootstrap、localStorage 永続化、focus / visibility refresh、バックグラウンド実行状態の再取得、終了時 cleanup は `src/lib/useAppLifecycle.ts` に集約する。
- `controllersRef` や `shellControllersRef` を abort する cleanup は unmount 時だけに限定する。
- rerender 前 cleanup で controller を abort すると、実行直後に「停止しました」へ誤遷移し、実行中表示や進行バーも消える。

### 8.4 間引きルール

間引いてよいもの:

- `assistant.reasoning_delta` のような高頻度で意味の薄い内部進捗。
- ログモーダルを開いていない時の巨大なコピー用テキスト生成。
- 非表示ペインや非表示タブの詳細ログ整形。
- 実行中の localStorage 保存頻度。

間引かないもの:

- ユーザー入力中のテキスト、カーソル、選択状態。
- CLI / モデル / 推論レベルなど、ユーザーが直接選ぶ操作UI。
- assistant の最終回答本文、stderr、エラー、確認待ち、完了、停止。
- コマンド確認に出す「これから渡す情報」の構造化表示。

### 8.5 残る制約

- ブラウザタブが完全に破棄された場合、クライアント側の live stream は当然失われる。
- サーバー自体が落ちた場合は、実行状態の復元はできない。
- reasoning_delta を間引くため、ストリームログには内部進捗の全イベントは残らない。

### 8.6 次の改善候補（未着手）

- stream event を pane ごとにバッファリングし、一定間隔でまとめて React state に反映する。
- 表示中でない pane の詳細ログ描画をさらに遅延する。
- localStorage ではなく IndexedDB に履歴を逃がす。
- 長い streamEntries を仮想リスト化する。
- サーバー側で run event をリングバッファに保持し、クライアント再接続時に差分取得できるようにする。

## 9. TAKOエラー調査報告

### 9.1 先に結論

今回の失敗は 1 つの不具合ではなく、複数の条件が重なって起きている。

- **一次原因:** OS 制御を含む要求を Codex 標準モードで流し、Windows sandbox と衝突したこと。
- **増幅要因:** Git 管理外 workspace で git 系の操作が走ったこと、PowerShell 5.1 と非互換なコマンドが生成されたこと。
- **TAKO 側の本質的な問題:** 作業ディレクトリの修正ではなく、事前警告と失敗要約の不足。

### 9.2 調査で確認した事実

#### Codex 標準モードは `--full-auto` で起動している

`server/src/cliRunner.ts` では、Codex の標準モード時に `--full-auto`、max 時に `--dangerously-bypass-approvals-and-sandbox` を付けている。標準モードでは Windows sandbox 前提の実行になる。

#### local 実行の cwd は選択 workspace に固定されている

`server/src/cliRunner.ts` では、local 実行時の作業ディレクトリを選択中 workspace に固定し、Codex にも `-C <workspace>` を渡している。`not a git repository` は「cwd が TAKO 側でズレた」ことを示す証拠ではない。

#### 既存の専用要約は一部しかない

`src/lib/appCore.ts` に既にある専用要約:

- `fatal: detected dubious ownership in repository`
- `fatal: not a git repository`
- Codex の保護パス書き込み拒否
- Gemini の Plan Mode

今回問題になったが専用要約がなかったもの:

- `CreateProcessWithLogonW failed: 1056`
- `Format-Hex : パラメーター名 'Count' ...`
- `Exit code: 1` かつ出力が空の後続失敗

#### PowerShell 7 前提は安全ではない

`Format-Hex` に `-Count` が存在しない環境が確認された。PowerShell 7 系の記法をそのまま期待するのは危険。

### 9.3 エラー別の原因と対策

#### `windows sandbox: CreateProcessWithLogonW failed: 1056`

**原因:** OS 制御系の要求（VSCode の複数起動、ウィンドウ位置制御等）と Windows sandbox 制約が衝突。

**対策:**
- `1056` を `src/lib/appCore.ts` で専用要約する。
- プロンプトに OS 制御やウィンドウ制御の語が多い場合、Codex 標準モードでは失敗しやすい旨を事前 notice で出す。
- この種類の要求では、まず調査報告や実装計画の生成を優先させる導線を入れる。

#### `Exit code: 1` / `Output: 空`

**原因:** 応答本文が空で終了コードが非 0 の場合に、直前の一次障害との関係が見えない形で失敗が表示される。

**対策:**
- 直前の `status`、`tool`、`stderr` を短く保持し、空応答失敗時にまとめて表示する。
- 「sandbox 失敗の後続である可能性」などの補助文を付ける。

#### `fatal: not a git repository`

**原因:** 対象 workspace に `.git` が存在しない状態で git コマンドが実行された。cwd バグではなく workspace と実行内容のミスマッチ。

**対策:**
- workspace 選択時または実行前に `.git` の有無を確認し、Git 管理外であることを notice 表示する。
- プロンプトに `git` や `commit` などが含まれる場合は、非 Git workspace だと失敗しやすいことを追加で警告する。

#### `Format-Hex : パラメーター名 'Count' に一致するパラメーターが見つかりません。`

**原因:** 生成された PowerShell コマンドが PowerShell 5.1 で解釈できなかった。

**対策:**
- `Format-Hex` の unsupported parameter を `src/lib/appCore.ts` で専用要約する。
- Windows 環境では、PowerShell 7 が確認できない限り PowerShell 5.1 互換を安全側の前提として扱う。

### 9.4 今回のエラーから見える TAKO 側の問題点

#### 失敗の事前警告が足りない

- OS 制御系タスク
- Git 管理外 workspace
- PowerShell バージョン差

これらは、実行前にかなりの確率で危険を察知できる。

#### 失敗の事後要約が足りない

- `not a git repository` は既に要約されているが、1056 と `Format-Hex -Count` は生の stderr に近い。
- 空応答失敗も、一次障害との関係が見えない。

#### 「実装」と「調査」の導線分岐が弱い

- OS 制御や VSCode 操作を含む要件でも、通常の workspace 編集と同じ感覚で実行される。
- その結果、標準 sandbox に不向きな要求でも、いきなり実行に進みやすい。

### 9.5 優先度付きの対策

| 優先度 | 対策 |
| --- | --- |
| 高 | `CreateProcessWithLogonW failed: 1056` の専用要約追加 |
| 高 | `Format-Hex -Count` 系互換性エラーの専用要約追加 |
| 高 | 非 Git workspace の事前 notice 追加 |
| 高 | 空応答失敗に直前 stderr と失敗フェーズを付ける |
| 中 | OS 制御系キーワードを検出したとき、標準モードでは調査優先で進める preflight を入れる |
| 中 | PowerShell / pwsh の利用可否を bootstrap 時に把握し、実行補助文へ反映する |
| 低 | プロンプト種別ごとに、実行より文書化を優先するテンプレートを増やす |
| 低 | 失敗ログの分類粒度をさらに細かくする |

### 9.6 最終結論

今回のエラーは、単一の TAKO バグではない。主因は、OS 制御色の強い要求を Codex 標準モードで処理しようとして Windows sandbox と衝突したことにある。そこへ、Git 管理外 workspace と PowerShell 互換性問題が重なり、さらに TAKO 側の事前警告と失敗要約が不足していたため、原因が分かりにくくなった。

TAKO 側でまず行うべき修正は、新しい VSCode 監視パネルの設計ではなく、以下の診断改善である。

- sandbox 失敗の専用要約
- PowerShell 互換性エラーの専用要約
- 非 Git workspace の事前警告
- 空応答失敗の文脈補強
